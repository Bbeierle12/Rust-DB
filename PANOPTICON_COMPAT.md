## asyncpg / SQLAlchemy / Alembic compatibility layer

The pgwire server (`src/bin/server.rs`) now includes two preprocessing stages that run before
every query reaches the engine — both for simple queries and extended (parameterized) queries.

### Stage 1 — `preprocess_sql`: strip `::type` casts

SQLAlchemy generates `$1::jsonb`, `'foo'::text`, `NULL::integer`, etc.
Rust-DB's SQL parser does not support PostgreSQL cast syntax.
`preprocess_sql` strips `::typename` from all SQL outside of string literals.
Handles compound types: `::double precision`, `::character varying`, `::time zone`.

### Stage 2 — `intercept_system_query`: virtual catalog responses

| Query pattern | Response |
|---|---|
| `SET …` / `RESET …` / `DISCARD …` / `DEALLOCATE …` | `SET` tag (no-op) |
| `SHOW …` | 1-row empty text result |
| `SELECT version()` | `"PostgreSQL 14.0 on Rust-DB (custom engine)"` |
| `SELECT current_schema()` | `"public"` |
| `SELECT current_database()` | `"rustdb"` |
| `SELECT current_setting(…)` | empty string |
| `SELECT 1` | `1` (INT8) |
| `pg_catalog.*` / `pg_type` / `pg_namespace` / `pg_class` / … | empty result |
| `information_schema.tables` | rows built from live catalog |
| `information_schema.tables` with `WHERE table_name = 'X'` | filtered rows |
| `SELECT EXISTS (SELECT FROM information_schema.tables WHERE …)` | `t` / `f` (BOOL) |
| `information_schema.columns` | rows built from live catalog with type mapping |
| any other `information_schema.*` | empty result |

### Engine fix — `CREATE TABLE IF NOT EXISTS`

`execute_create_table` in `src/engine/mod.rs` now checks the `if_not_exists` flag.
When the flag is set and the table already exists, returns `Execute(0)` instead of an error.
Required for Alembic migrations (`CREATE TABLE IF NOT EXISTS alembic_version …`).

## Known behavioural notes (not gaps)

### DEFAULT column constraints are silently ignored

`DEFAULT 1` on the `alerts.count` column is parsed successfully but the engine
does not enforce it. Every INSERT must supply a value for `count` explicitly.

### Table aliases in JOINs are ignored

`FROM devices d LEFT JOIN alerts a ON d.ip = a.device_ip` — the alias tokens
`d` and `a` are silently dropped by the parser stage; only the real table name
is used when resolving schemas. Column references in SELECT, WHERE, GROUP BY,
and ON must use the **full table name** (e.g. `devices.ip`, not `d.ip`).

### LEFT JOIN COUNT includes null-padded rows

`COUNT(*)` and `COUNT(col)` both map to `AggFunc::Count` which counts **every
row** including the null-padded rows produced by unmatched LEFT JOIN entries.
So a device with no alerts still gets `alert_count = 1` (not 0) after a LEFT
JOIN + GROUP BY. The Axum API bridge should subtract 1 when all `alerts.*`
columns are NULL, or use a post-processing step.

### Aggregate column name for COUNT(*) without alias

`SELECT COUNT(*) FROM table` produces a result column named `"count"` (the
lower-cased function name). Use `SELECT COUNT(*) AS n FROM table` for a
predictable alias.
